## The Milestone

500 problems. It sounds like a number worth celebrating, and it is — but not for the reasons most people think.

The milestone isn't about the count. It's about the **mental models** that accumulate underneath it. The moment you stop seeing a problem as a puzzle and start seeing it as a pattern you've solved before in a different costume.

## The First 100: Confusion

The first 100 problems were humbling. Easy problems felt medium. Medium problems felt hard. I was brute-forcing everything, not understanding *why* a solution worked — just that it passed.

```java
// Early me: just make it work
public int[] twoSum(int[] nums, int target) {
    for (int i = 0; i < nums.length; i++) {
        for (int j = i + 1; j < nums.length; j++) {
            if (nums[i] + nums[j] == target) return new int[]{i, j};
        }
    }
    return new int[]{};
}
```

O(n²). It works. But it's not thinking.

## Problems 100–300: Pattern Recognition Kicks In

This is where the investment starts paying returns. You begin to *feel* when something is a sliding window problem. When the recursive structure screams dynamic programming. When you should reach for a monotonic stack.

```java
// Later me: recognize the HashMap pattern
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (seen.containsKey(complement)) return new int[]{seen.get(complement), i};
        seen.put(nums[i], i);
    }
    return new int[]{};
}
```

O(n). One pass. Space traded for time, and now you *understand* why that's worth it.

## The Patterns That Showed Up Everywhere

After 500 problems, certain patterns are so ingrained they feel like reflexes:

**Two Pointers** — Any array problem asking about pairs, subarrays, or sorted relationships. Start from both ends and converge.

**Sliding Window** — Any "maximum/minimum subarray of size k" or "longest substring with constraint" problem. Expand right, shrink left.

**DFS/BFS on Graphs** — Islands, connected components, shortest paths. The template barely changes, only the state tracking does.

**Dynamic Programming** — If you're making decisions at each step and the future depends on current state, it's probably DP. Define the state clearly before writing a single line.

**Monotonic Stack** — Next greater element, largest rectangle, daily temperatures. Keep a decreasing stack, pop when violated.

## What Competitive Programming (TECH-SPARDA) Added

Winning 2nd at the inter-NIT coding competition before deep LeetCode practice felt different from LeetCode practice itself. Competition forces speed *and* correctness simultaneously. No multiple attempts. No hints. No editorial after 20 minutes.

It trained the instinct to estimate complexity before coding. To choose the approach in your head before your fingers touch the keyboard.

## The Honest Advice

**Don't grind randomly.** Topic-focused practice beats random mediums every time. Spend a week only on trees. Then graphs. Then DP. The connections between topics appear much faster this way.

**Read others' solutions.** Not to copy — to see how someone else's mind structured the same problem. Sometimes a 3-line solution to a problem you solved in 40 lines teaches more than the next 10 problems.

**Time yourself.** Unlimited time on LeetCode is comfortable but misleading. Real interviews have clocks.

## Problem 500

I don't remember which problem it was. That's the point. After 500, individual problems stop mattering. The way of seeing the problems is what stays.
